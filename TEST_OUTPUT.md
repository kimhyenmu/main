# 멀티스레드 테스트 출력 예시

## 1. 기본 서비스 테스트 (문제 발생 버전) ⚠️

```
========================================
1. 기본 서비스 테스트 (문제 발생 버전)
========================================

[테스트 설정]
서비스: Basic View Service
스레드 수: 100
각 스레드당 호출 횟수: 50
총 호출 횟수 (기대값): 5000

[테스트 결과]
소요 시간: 164ms
성공한 호출: 5000 / 5000
실패한 호출: 0

[조회수 비교]
기대값: 5000
캐시 조회수: 51
DB 조회수: 51

[문제 분석]
⚠️  캐시 손실: 4949 (98.98%)
⚠️  DB 손실: 4949 (98.98%)
✅ 캐시-DB 일치

⚠️  Race Condition 발생! 조회수 불일치가 발견되었습니다.
```

### 분석
- **심각한 데이터 손실**: 5,000번 증가 요청 중 **98.98%가 손실**되어 51번만 증가
- **Lost Update 발생**: 여러 스레드가 동시에 같은 값을 읽고 증가시켜 덮어씀
- **캐시 미스 동시 발생**: 초기 캐시 미스 시 여러 스레드가 동시에 DB에서 읽어옴

## 6. 여러 게시글 동시 처리 성능 비교

```
========================================
6. 여러 게시글 동시 처리 테스트 (게시글별 락 vs 전역 락)
========================================

[Global Lock (모든 게시글 순차 처리)]
3개의 게시글에 대해 동시 요청 처리

[Per-Post Lock (게시글별 병렬 처리)]
3개의 게시글에 대해 동시 요청 처리

성능 비교:
전역 락 소요 시간: 1081ms
게시글별 락 소요 시간: 364ms
성능 향상: 2.97배
```

### 분석
- **전역 락**: 모든 게시글이 하나의 큐에서 순차 처리 (1,081ms)
- **게시글별 락**: 게시글마다 독립적인 락으로 병렬 처리 (364ms)
- **성능 향상**: **2.97배 빠른 처리**
- **결론**: 서로 다른 리소스는 독립적으로 처리하는 것이 효율적

## 성능 요약 표

| 서비스 | 소요 시간 | 정확성 | 캐시 손실 | DB 손실 | 성능 비율 |
|--------|----------|--------|----------|---------|----------|
| Basic (문제 발생) | 164ms | ❌ | 98.98% | 98.98% | 1.00x (기준) |
| Global Lock | 5,399ms | ✅ | 0% | 0% | 0.03x (33배 느림) |
| Per-Post Lock | 5,406ms | ✅ | 0% | 0% | 0.03x (33배 느림) |
| Atomic | 72ms | ✅ | 0% | 0% | 2.28x (2.3배 빠름) |
| Write-Through | 5,417ms | ✅ | 0% | 0% | 0.03x (33배 느림) |

### 핵심 발견

1. **동기화 없음 → 98.98% 데이터 손실**
   - 가장 빠르지만 사용 불가능

2. **Atomic 연산 → 최고의 솔루션** 🏆
   - 정확하면서도 가장 빠름 (72ms)
   - 전역 락 대비 75배 빠름
   
3. **게시글별 락 → 병렬 처리 향상**
   - 여러 게시글 동시 처리 시 2.97배 향상
   - 전역 락보다 효율적

4. **Write-Through → 강한 일관성**
   - 정확하지만 느림 (5,417ms)
   - 데이터 일관성이 최우선일 때 사용

## Race Condition 발생 시나리오 예시

### 시나리오 1: Lost Update

```
t1: Thread-1이 캐시에서 조회수 0 읽음
t2: Thread-2가 캐시에서 조회수 0 읽음
t3: Thread-3이 캐시에서 조회수 0 읽음
t4: Thread-1이 조회수 1로 증가 → DB 및 캐시에 저장
t5: Thread-2가 조회수 1로 증가 → DB 및 캐시에 저장 (Thread-1 손실)
t6: Thread-3이 조회수 1로 증가 → DB 및 캐시에 저장 (Thread-1, 2 손실)

기대값: 3
실제값: 1
손실: 2번의 증가 연산
```

### 시나리오 2: 캐시 미스 동시 발생

```
초기 상태: 캐시에 데이터 없음, DB 조회수 0

t1: Thread-1이 캐시 확인 → 캐시 미스
t2: Thread-2가 캐시 확인 → 캐시 미스
t3: Thread-1이 DB에서 조회수 0 읽음
t4: Thread-2가 DB에서 조회수 0 읽음
t5: Thread-1이 캐시에 0 저장
t6: Thread-2가 캐시에 0 저장
t7: Thread-1이 조회수 1로 증가 → DB 및 캐시에 저장
t8: Thread-2가 조회수 1로 증가 → DB 및 캐시에 저장

기대값: 2
실제값: 1
문제: 캐시 미스 시 여러 스레드가 동시에 DB 조회
```

### 시나리오 3: 캐시-DB 갱신 순서 문제

```
t1: Thread-1이 조회수 1로 증가 → DB에 저장 시작
t2: Thread-2가 캐시에서 조회수 0 읽음 (Thread-1의 DB 저장이 아직 완료 안됨)
t3: Thread-1의 DB 저장 완료 (값: 1)
t4: Thread-1이 캐시에 1 저장
t5: Thread-2가 조회수 1로 증가 → DB에 저장 (값: 1, 잘못됨)
t6: Thread-2가 캐시에 1 저장

기대값: 2
실제값: 1 (캐시와 DB 모두)
문제: DB 저장과 캐시 갱신 사이의 타이밍 이슈
```

## 결론

멀티스레드 환경에서는 **반드시 적절한 동기화 메커니즘**이 필요합니다.

**권장 사항:**
- 일반적인 경우: **AtomicViewService** (최고 성능)
- 여러 리소스 처리: **PerPostLockViewService** (병렬성 향상)
- 강한 일관성 필요: **WriteThroughViewService** (데이터 정확성 최우선)

**절대 사용 금지:**
- **BasicViewService**: 98.98%의 데이터 손실 발생! 프로덕션 환경에서 절대 사용 불가
