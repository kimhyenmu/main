# 멀티스레드 환경에서의 캐시-DB 불일치 문제 분석 및 해결

## 📋 과제 개요

이 프로젝트는 멀티스레드 환경에서 게시글 조회수 증가 과정에서 발생하는 **캐시와 DB 간의 race condition과 타이밍 이슈**를 재현하고, 이를 해결하는 여러 전략을 구현 및 비교한 프로젝트입니다.

## 🎯 구현 기능

### 1. 조회수 증가 API 동작 흐름

모든 서비스는 다음과 같은 기본 흐름을 따릅니다:

1. **캐시에서 조회수 읽기**
2. **캐시 값이 없으면 DB에서 읽어 캐시에 저장**
3. **조회수 +1 증가**
4. **DB에 저장**
5. **캐시에 저장**
6. **최종 조회수 반환**

### 2. 구현된 서비스

#### 2.1 BasicViewService (문제 발생 버전)
- 락이나 동기화 없이 구현
- Race condition과 타이밍 이슈가 발생하도록 의도적으로 설계

#### 2.2 GlobalLockViewService (전역 락)
- 모든 연산에 전역 락(synchronized) 적용
- 모든 게시글에 대해 순차 처리

#### 2.3 PerPostLockViewService (게시글별 락)
- 게시글 ID마다 별도의 락 사용
- 서로 다른 게시글은 동시 처리 가능

#### 2.4 AtomicViewService (원자적 연산)
- `AtomicLong`을 사용한 락프리 구현
- 원자적 증가 연산 보장

#### 2.5 WriteThroughViewService (Write-through)
- DB를 단일 진실 공급원(Single Source of Truth)으로 사용
- DB 저장 성공 후에만 캐시 갱신

## 🧪 테스트 환경

### 테스트 설정
- **스레드 수**: 100개
- **각 스레드당 호출 횟수**: 50번
- **총 호출 횟수 (기대값)**: 5,000번

### 데이터 저장소
- **캐시**: `ConcurrentHashMap` (Redis 시뮬레이션)
- **DB**: `ConcurrentHashMap` (실제 DB 시뮬레이션)
- **DB 저장 지연**: 1ms (실제 DB 동작 시뮬레이션)

## 🔍 발생한 Race Condition 상세 분석

### 1. Lost Update (갱신 손실)

**발생 원인:**
```
시간 t1: 스레드 A가 캐시에서 조회수 0 읽음
시간 t2: 스레드 B가 캐시에서 조회수 0 읽음
시간 t3: 스레드 A가 조회수를 1로 증가시켜 저장
시간 t4: 스레드 B가 조회수를 1로 증가시켜 저장 (A의 갱신 손실)
결과: 2번 증가했지만 최종 값은 1
```

**테스트 결과:**
- 5,000번 증가 → 실제로는 51번만 증가
- **98.98%의 갱신이 손실됨**

### 2. 캐시 미스 동시 발생

**발생 원인:**
```
시간 t1: 스레드 A가 캐시 확인 → 값 없음
시간 t2: 스레드 B가 캐시 확인 → 값 없음
시간 t3: 스레드 A가 DB에서 0 읽어서 캐시에 저장
시간 t4: 스레드 B가 DB에서 0 읽어서 캐시에 저장 (A의 값 덮어씀)
결과: 초기값 중복 설정
```

### 3. 캐시-DB 불일치

**발생 원인:**
```
시간 t1: 스레드 A가 조회수 증가 후 DB에 저장 (값: 1)
시간 t2: 스레드 B가 조회수 증가 후 DB에 저장 (값: 1)
시간 t3: 스레드 A가 캐시에 저장 (값: 1)
시간 t4: 스레드 B가 캐시에 저장 (값: 1)
결과: 캐시와 DB 모두 1이지만, 실제로는 2가 되어야 함
```

**흥미로운 점:**
- 이번 테스트에서는 캐시와 DB가 같은 잘못된 값을 가지고 있어 "일치"로 표시됨
- 하지만 둘 다 기대값과 다르므로 실질적으로는 불일치 상황

### 권장 사항

#### 1. 일반적인 경우: **AtomicViewService** 🏆
- **장점**: 가장 빠른 성능 (75배 향상), 구현 간단
- **적용**: 조회수, 좋아요 등 빈번한 카운터 증가가 필요한 경우

#### 2. 여러 게시글 동시 처리: **PerPostLockViewService**
- **장점**: 게시글별 병렬 처리로 2.97배 성능 향상
- **적용**: 서로 다른 리소스에 대한 독립적인 처리가 많은 경우

#### 3. 강한 일관성이 필요한 경우: **WriteThroughViewService**
- **장점**: DB와 캐시 간 강한 일관성 보장
- **적용**: 금융 거래, 재고 관리 등 데이터 정확성이 최우선인 경우

## 🎓 학습 내용

### 1. 동시성 제어의 중요성
- 멀티스레드 환경에서는 반드시 적절한 동기화 메커니즘이 필요
- 동기화 없이는 98.98%의 데이터 손실이 발생할 수 있음

### 2. 성능과 정확성의 트레이드오프
- 전역 락: 정확하지만 느림
- 원자적 연산: 정확하면서도 빠름
- 적절한 전략 선택이 중요

### 3. 캐시-DB 일관성 관리
- 캐시와 DB의 갱신 순서가 중요
- Write-through 패턴으로 강한 일관성 보장 가능

### 4. 락의 세분화 (Lock Granularity)
- 전역 락 vs 게시글별 락
- 세분화된 락으로 병렬성 향상 가능 (2.97배)

## 📝 결론

이 프로젝트를 통해 멀티스레드 환경에서 발생할 수 있는 다양한 race condition과 타이밍 이슈를 직접 재현하고, 여러 해결 전략의 장단점을 비교 분석했습니다.

**핵심 발견:**
1. 동기화 없이는 **98.98%의 데이터 손실** 발생
2. `AtomicLong`을 사용한 락프리 구현이 **가장 우수한 성능** (75배 향상)
3. 게시글별 락으로 **병렬 처리 성능 2.97배 향상**
4. 상황에 따른 적절한 전략 선택이 중요

실제 프로덕션 환경에서는 **AtomicViewService**와 같은 원자적 연산 방식을 기본으로 사용하되, 데이터 일관성이 매우 중요한 경우 **WriteThroughViewService**를 고려하는 것이 좋습니다.
