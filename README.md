# 트랜잭션 격리 수준과 락 실습

이 실습은 데이터베이스 트랜잭션의 격리 수준과 락 메커니즘을 학습하기 위한 종합적인 실습 환경입니다.

## 실습 구성

### 1. 데이터베이스 설정
- **파일**: `database_setup.py`
- **기능**: SQLite 데이터베이스 초기화 및 테이블 생성
- **테이블**: seats, reservations, transaction_log

### 2. 격리 수준 테스트
- **파일**: `isolation_level_tests.py`
- **내용**:
  - Read Uncommitted 시뮬레이션
  - Read Committed 시뮬레이션  
  - Phantom Read 시뮬레이션
  - Serializable 격리 수준 테스트

### 3. 락 비교 실험
- **파일**: `lock_comparison.py`
- **내용**:
  - 테이블 레벨 락 vs 행 단위 락
  - 락 타임아웃 테스트
  - 데드락 시뮬레이션

### 4. 좌석 예매 시스템
- **파일**: `seat_booking_system.py`, `concurrent_booking_test.py`, `simple_concurrent_test.py`
- **기능**:
  - 동시성 제어를 통한 안전한 좌석 예매
  - 비관적 락과 낙관적 락 비교
  - 경합 조건 시뮬레이션

### 5. 종합 데모
- **파일**: `comprehensive_demo.py`
- **기능**: 모든 실습 내용을 하나로 통합한 종합 데모

## 실행 방법

```bash
python3 comprehensive_demo.py
```

## 실습 결과 요약

### 1. 격리 수준별 동시성 문제

#### Non-repeatable Read
- **발생 조건**: 같은 트랜잭션 내에서 같은 데이터를 여러 번 읽을 때
- **결과**: 다른 트랜잭션의 커밋으로 인해 읽기 결과가 달라짐
- **실습 결과**: ✅ 성공적으로 재현됨

#### Phantom Read  
- **발생 조건**: 범위 쿼리(COUNT, SELECT WHERE 조건)를 반복할 때
- **결과**: 새로운 행이 추가되거나 삭제되어 결과 집합이 변경됨
- **실습 결과**: ✅ 성공적으로 재현됨

### 2. 락 메커니즘

#### 테이블 레벨 락 (SQLite 기본)
- **특징**: 전체 테이블에 대한 배타적 접근
- **장점**: 구현이 간단하고 데드락 위험이 낮음
- **단점**: 동시성이 제한됨

#### 락 타임아웃
- **동작**: 설정된 시간 내에 락을 획득하지 못하면 오류 발생
- **실습 결과**: 5초 타임아웃 설정으로 "database is locked" 오류 확인

### 3. 좌석 예매 시스템 동시성 제어

#### 경합 조건 (Race Condition)
```
시나리오: 3명이 동시에 마지막 좌석 1개 예매 시도
결과: 1명 성공, 2명 실패 - ✅ 올바른 동작
```

#### 비관적 락 (Pessimistic Locking)
```
전략: BEGIN IMMEDIATE로 즉시 배타적 락 획득
결과: 순차적 처리로 안전한 예매 보장
성능: 락 대기 시간으로 인한 성능 저하
```

#### 낙관적 락 (Optimistic Locking)
```
전략: 버전 필드를 이용한 Compare-And-Swap
장점: 락 대기 시간 없음
단점: 충돌 시 재시도 필요
```

## 핵심 학습 내용

### 1. 데이터 일관성 보장 조건
- ✅ 좌석이 1개만 남은 상황에서 정확히 1명만 예매 성공
- ✅ 실패한 사용자는 명확한 실패 응답 수신
- ✅ 중복 확정이나 초과 판매(Oversell) 방지
- ✅ 트랜잭션 종료 후 데이터 정합성 유지

### 2. 동시성 제어 전략 비교

| 전략 | 장점 | 단점 | 사용 사례 |
|------|------|------|-----------|
| 비관적 락 | 안전성 높음 | 성능 저하 | 충돌 가능성이 높은 경우 |
| 낙관적 락 | 성능 우수 | 재시도 필요 | 충돌 가능성이 낮은 경우 |
| Serializable | 완전한 격리 | 동시성 제한 | 강한 일관성이 필요한 경우 |

### 3. 실제 환경 적용 고려사항
- **트랜잭션 범위 최소화**: 락 보유 시간을 줄여 성능 향상
- **타임아웃 설정**: 무한 대기 방지를 위한 적절한 타임아웃 설정
- **재시도 로직**: 충돌 시 적절한 재시도 전략 구현
- **모니터링**: 락 대기 시간과 충돌 빈도 모니터링

## 환경 정보
- **데이터베이스**: SQLite 3 (Python sqlite3 모듈)
- **언어**: Python 3.13.3
- **동시성**: threading 모듈 사용
- **격리 수준**: SQLite 기본 (Serializable)

## 주의사항
- SQLite는 행 레벨 락을 지원하지 않으므로 테이블 레벨 락으로 시뮬레이션
- 실제 운영 환경에서는 PostgreSQL, MySQL 등의 RDBMS 사용 권장
- 네트워크 지연, 디스크 I/O 등 실제 환경의 변수들을 고려해야 함
